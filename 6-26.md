Flujo actual
Estados y actores

DRAFT → PENDING_REVIEW → APPROVED → DESIGN_IN_PROGRESS → PENDING_ADVERTISER_APPROVAL → READY_TO_LAUNCH → LAUNCHED
                      ↘ REJECTED        ↑ (auto al guardar draft)    ↘ CHANGES_REQUESTED ↗
Paso a paso
1. Commercial crea la solicitud (DRAFT)

POST /branding-requests — crea con marca, presupuesto, juego elegido
Sube recursos corporativos (logos, imágenes de referencia) al prefijo private/ de R2
Actualiza targeting: categorías, género, edad, municipios, goal de campaña — PATCH /branding-requests/{id}/config
POST /branding-requests/{id}/submit → PENDING_REVIEW
2. Admin revisa

Ve la solicitud, lee la descripción de marca y los recursos corporativos
POST /admin/branding-requests/{id}/approve → asigna designer + escribe adminNotes + pasa a APPROVED
O reject → REJECTED con adminNotes explicando por qué
3. Designer recibe y trabaja

La solicitud aparece en GET /game-designers/me/branding-requests
El primer PATCH /game-designers/me/branding-requests/{id}/draft (guardar progreso) auto-transiciona a DESIGN_IN_PROGRESS
Sube assets al CDN público (imágenes, audio, video del juego), confirma cada uno
Itera: guarda draft, sube assets, repite hasta que el diseño esté listo
PATCH /game-designers/me/branding-requests/{id}/notes — puede escribir en designerNotes
POST /game-designers/me/branding-requests/{id}/submit-design → el draftFormData se limpia (strip de assetIds) y se guarda como gameConfig → PENDING_ADVERTISER_APPROVAL
4. Commercial revisa el diseño

POST /branding-requests/{id}/approve-design → READY_TO_LAUNCH
POST /branding-requests/{id}/request-design-changes → escribe en designerNotes qué cambiar → CHANGES_REQUESTED
El designer ve el campo designerNotes en el detalle y vuelve a trabajar
5. Admin lanza

Lanza la campaña desde el admin → LAUNCHED (crea Campaign)
Cómo funcionan las notas hoy
Campo	Quién escribe	Cuándo	Quién lee
adminNotes	Admin	Al aprobar o rechazar	Commercial (y designer lo ve en su detalle)
designerNotes	Commercial	Al pedir cambios (request-design-changes)	Designer
designerNotes	Designer	Vía PATCH /notes	Commercial (pero sobrescribe las del commercial)
El problema: designerNotes es el mismo campo para dos propósitos distintos — cuando el commercial escribe "cambia el color del botón" y luego el designer escribe su respuesta, la nota del commercial desaparece.

Qué cambiaría
Los problemas principales son tres:

1. El campo designerNotes es ambiguo y se sobreescribe
Fix: Separar en dos campos distintos en BrandingRequest:


// Ya existe — notas del admin al commercial
@Column(name = "admin_notes", length = 1000)
private String adminNotes;

// NUEVO — lo que el commercial le pide cambiar al designer
@Column(name = "change_request_notes", length = 2000)
private String changeRequestNotes;

// NUEVO — observaciones técnicas del designer al commercial
@Column(name = "designer_feedback", length = 2000)
private String designerFeedback;
El endpoint request-design-changes escribe en changeRequestNotes. El endpoint PATCH /notes escribe en designerFeedback. El DesignerBrandingDetailDTO expone changeRequestNotes (para que el designer sepa qué cambiar). El commercial ve designerFeedback en su detalle.

2. No hay notificaciones — todo el flujo es silencioso
Actualmente ninguna transición de estado envía email. El designer no sabe que le asignaron un proyecto. El commercial no sabe cuando el designer entregó. El designer no sabe cuando le piden cambios.

Fix: Agregar @Async email en cada transición clave:

Evento	Destinatario	Contenido
APPROVED	Designer	"Te asignaron el proyecto X de la marca Y — resumen del brief"
PENDING_ADVERTISER_APPROVAL	Commercial	"El diseñador entregó tu juego — entra a revisarlo"
CHANGES_REQUESTED	Designer	"El anunciante pidió cambios: {changeRequestNotes}"
READY_TO_LAUNCH	Admin	"La campaña X está lista para lanzar"
REJECTED	Commercial	"Tu solicitud fue rechazada: {adminNotes}"
3. El commercial aprueba/rechaza a ciegas — no puede ver el diseño
Cuando el estado es PENDING_ADVERTISER_APPROVAL, el BrandingRequestDetailDTO del commercial no expone el gameConfig que entregó el designer. El commercial no sabe qué está aprobando.

Fix: Exponer el gameConfig en el detalle del commercial cuando el estado es PENDING_ADVERTISER_APPROVAL o posterior. Idealmente también una URL de preview del juego ya configurado.

Mejora menor adicional: historial de revisiones
Si el commercial pide cambios varias veces, solo queda el último changeRequestNotes. Para proyectos con múltiples iteraciones conviene una tabla design_revision:


@Entity
public class DesignRevision {
    private Long id;
    private BrandingRequest brandingRequest;
    private String notes;          // qué pidió el commercial
    private int revisionNumber;
    private ZonedDateTime createdAt;
}
Esto es opcional y agrega complejidad — solo lo implementaría si el negocio espera proyectos con más de 2-3 iteraciones frecuentes.
